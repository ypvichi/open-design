// part 02 · workflow map

From a one-line prompt to a shipped file in four beats

①
Take the room's prompt

Ask the audience for a real ask — a landing page, a dashboard, a pitch deck — and type exactly what they say.

②
Lock the direction

One click sets palette, typography, and spacing so every following screen stays visually consistent.

③
Generate on screen

The agent writes real, runnable files in the sandbox — the audience watches code compile, not a loading spinner.

④
Hand it over, on the spot

Drop the file into a repo or a Slack channel before the session ends — no "we'll follow up" needed.

This is the workflow map — four beats, and I run through all four live in every session.

Beat one, take the room's prompt. Not a warm-up example — the actual thing the audience says they need, typed verbatim.

Beat two, lock the direction. One click sets the palette, fonts, and spacing, so everything the agent generates after this stays visually consistent — no per-screen tuning live on stage.

Beat three, generate on screen — the moment that sells the room. The agent reads full context and writes real, runnable files in the sandbox, and the audience watches it happen in real time, not a spinner cutting to a finished screenshot.

Beat four, hand it over on the spot. The file goes to a repo or a Slack channel before we move to Q&A — the prospect leaves with something real, not a follow-up email.

// part 03 · pilot & risk control

Pilot format & risk controls

01
🎯 Pilot scope: one session, one live prompt

30 minutes, one audience-chosen problem, one agent, one deliverable — repeatable across prospects, not a one-off show.

02
🛡️ Risk control: it runs without the internet

Local-first, so a flaky venue Wi-Fi never stalls the demo; a rehearsed backup prompt is ready if the live one runs long.

03
📊 Success bar: they can name the file

Pass/fail signal: can the audience say what got built and where it lives — a repo link, a preview URL — by the time we close?

Now for the part that gets this funded internally — how we scope the pilot and control the risk of running it live.

First, pilot scope. One session, 30 minutes, one problem the audience picks in the room — not five feature tours crammed into one slot. That's what makes it repeatable across ten prospects a month instead of a single showcase event.

Second, risk control. Because Open Design runs locally, a bad venue Wi-Fi connection never stalls the live moment — the agent and sandbox are on the laptop. If the live prompt runs long, I have one rehearsed backup prompt ready that still lands inside the 30 minutes.

Third, the success bar — and this is the metric that matters to a DevRel manager: can the audience, unprompted, say what file got built and where it lives? If yes, the session earned a follow-up meeting. If no, we adjust the pilot script, not the product.

Put those three together and you have a pilot you can run weekly, not once. Now, the numbers that make the case to fund it.

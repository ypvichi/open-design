# Studio Chat Assistant 回复信息架构

> 关联方案：[`studio-chat-visual-discovery.md`](./studio-chat-visual-discovery.md)

## 1. 现在的问题

### 问题总结

当前 Assistant 回复更接近 Agent 运行事件的逐条回放：解释文字、思考、工具调用、文件变化、状态和最终总结按时间顺序连续出现。信息是完整的，但用户需要自己完成“哪些内容属于同一个任务、现在做到哪里、最终结果是什么”的聚合。

核心问题包括：

1. **任务边界不清晰**：一次用户请求可能包含多个子任务，但回复没有稳定分组。
2. **过程信息权重过高**：运行细节与结果占用相近空间，长任务难以扫描。
3. **完成后没有收束**：任务结束后仍保留完整展开的过程，持续占据 Chat 高度。
4. **重要状态缺少优先级**：错误、审批、产物和下一步可能与普通过程信息混排。
5. **简单任务也显得复杂**：短任务被多层卡片和状态包装后，反而降低阅读效率。

## 2. 目标

### 2.1 从事件流水转为任务视图

用户首先看到任务及其状态，再按需查看任务内部发生的工具调用和文件操作。

### 2.2 运行中提供足够反馈，但不过度暴露

运行中展示当前任务、最新关键进度和需要用户关注的状态；重复、低价值过程默认合并。

### 2.3 完成后突出结果

任务结束后，Chat 优先展示：

- 做完了什么。
- 产出了什么。
- 哪些重要决策被采用。
- 用户接下来可以做什么。

过程信息收进可展开详情。

### 2.4 保持异常和用户操作可见

错误、审批、登录、权限、冲突和需要用户确认的内容不参与普通折叠，避免“界面更简洁”以牺牲可恢复性为代价。

## 3. 展示原则

1. **结果优先于过程**。
2. **用户语言优先于工具语言**。
3. **当前状态优先于历史状态**。
4. **异常优先于普通进度**。
5. **短任务保持短，复杂任务才分组**。
6. **默认收起不等于删除，过程始终可回看**。

## 4. 回复结构

### 4.1 Assistant 回合

一个 Assistant 回合仍然对应一次 Agent run，但展示上分成：

1. **回合摘要**：对用户请求的直接回应或最终结论。
2. **任务组**：一个或多个用户可理解的执行阶段。
3. **产物与结果**：文件、预览、数据或结论。
4. **异常与待操作**：错误、审批、授权和恢复入口。
5. **下一步**：只在回合完成且没有待回答问题、异常或审批时出现。

### 4.2 任务组

任务组不是按工具名称分组，而是按用户目标分组。例如：

- “分析现有页面”
- “重构导航结构”
- “生成视觉稿”
- “验证构建结果”

每个任务组包含：

- 任务名称。
- 状态：等待、运行中、成功、失败或取消。
- 一句话进度或结果。
- 相关工具调用与文件操作。
- 可选的耗时和产物数量。

第一阶段可以基于现有 TodoWrite、工具序列和文件操作推导任务组，不要求立即修改 Agent 协议。

## 5. 不同状态下的体验

### 5.1 运行中

默认展示：

- 当前正在执行的任务。
- 最近一条有意义的进度。
- 已完成任务的紧凑状态。
- 错误、审批或待用户操作。

默认收起：

- 连续重复的读取、搜索和检查事件。
- 对用户决策没有影响的内部工具参数。
- 已经被后续状态替代的中间状态。

### 5.2 完成

- 回合顶部或底部给出简洁结果摘要。
- 已完成任务折叠为一行或紧凑列表。
- 产物、预览和主要操作保持可见。
- 用户可以展开任一任务查看完整过程。
- 下一步推荐只出现一次，不与问题卡或错误恢复竞争。

### 5.3 失败

- 失败任务保持展开。
- 明确失败发生在哪个任务，而不是只显示 run failed。
- 展示用户可执行的恢复动作。
- 已成功的前置任务仍然保留完成状态，不把整个回合模糊成失败。

### 5.4 等待用户

- 问题卡、审批或授权操作成为唯一主动作。
- 普通任务进度暂停或降权。
- 不展示“继续处理”“生成产物”等完成态推荐。

### 5.5 首次产物满意度回访

在用户首次看到成功生成的可预览产物后，可以展示一次轻量满意度模块，用于判断首轮交付是否达到预期。它是结果反馈，不是每次 run 结束后的固定步骤。

仅在同时满足以下条件时展示：

- 当前项目第一次成功生成可预览产物。
- 产物已经可以在 Workspace 打开或预览。
- 当前没有待回答问题、错误、审批或授权。
- 该项目此前没有展示过满意度模块。

以下场景不展示：

- discovery、纯问答、失败、重试或未产生可预览产物的 run。
- 对已有产物的小修改、重新生成和后续迭代。
- 用户已经回答、跳过或关闭过该模块。

交互采用“未达到预期 / 一般 / 满意”三个轻量选项；用户选择后可按需补充原因，但不要求填写。模块在视觉上低于产物和继续操作，不阻断 Chat，也不替代正常的下一步操作。

展示状态需要按项目持久化，不能只保存在当前页面内存中。刷新、重新进入项目或切换设备后都不应再次出现；跳过或关闭同样视为已经展示。

## 6. 聚合与收起规则

### 6.1 应聚合

- 同一任务下连续发生的多个工具调用。
- 多次读取、搜索、检查和状态更新。
- 对同一文件的连续编辑。
- 已经完成且没有异常的执行阶段。

### 6.2 不应聚合

- 只有一两句文字的简单回答。
- 需要用户阅读的设计判断、比较和建议。
- 错误、审批、权限和安全提示。
- 用户明确要求查看的日志或调试过程。
- 彼此独立、结果不同的多个任务。

### 6.3 默认展开策略

- 当前运行任务：展开。
- 最近失败任务：展开。
- 等待用户操作的任务：展开。
- 已完成任务：收起。
- 取消任务：紧凑展示，可展开查看原因。

## 7. 与视觉化提问的关系

两个方案在同一条 Chat 时间线上按照状态衔接：

1. Assistant 提出问题时，只展示内联问题卡。
2. 用户 Submit 或 Skip 后，问题卡折叠为回答摘要。
3. Agent 开始执行后，出现运行中的任务组。
4. 任务完成后，过程收起并突出结果与产物。
5. 确认没有待回答问题、错误或审批后，才展示下一步推荐。

问题卡不属于任务组，也不能被任务聚合隐藏；它是一次明确的用户决策节点。

## 8. 前端范围

第一阶段涉及：

- `AssistantMessage` 的回合级布局。
- Todo、工具组、文件操作与状态事件的任务归属。
- 运行中与完成态的展开策略。
- 任务组标题、状态、摘要和详情交互。
- 错误、审批、问题卡与下一步的优先级。
- 首次产物满意度模块及其一次性展示状态。
- 窄 Chat 列和长任务的滚动体验。

第一阶段不涉及：

- 改写 Agent 的规划能力。
- 新的 run / tool 协议。
- 自动总结质量模型。
- 跨会话任务图或项目级历史看板。

## 9. 数据验证与埋点

需要建立以下基线和事件：

- 任务组曝光、展开和再次收起。
- 完成态默认高度、回合总高度和滚动距离。
- 产物打开、错误恢复、审批完成和下一步点击。
- 首次产物满意度模块的曝光、回答、跳过、关闭和可选原因。
- 用户在完成后重新展开过程的比例。
- 长任务与短任务分别统计，避免平均值掩盖差异。

重点判断：

- 默认摘要是否真的够用。
- 哪些任务类型最常被展开。
- 收起过程后是否影响错误恢复和产物发现。
- 回复变短是否提升后续迭代率，而不是只减少可见信息。
- 满意度模块是否影响产物打开和后续迭代；满意度结果只作为体验诊断，不单独作为成功指标。

新增埋点需要先对照现行 analytics contracts 与埋点文档，不在方案阶段凭空新增事件命名。

## 10. 分阶段交付

### 第一阶段：静态原型

- 使用录制的真实 run 数据验证任务分组和状态层级。
- 覆盖短回答、长任务、多任务、失败和审批场景。
- 确认展开、收起和滚动行为。

### 第二阶段：现有事件推导

- 基于 TodoWrite、工具组和文件操作建立任务分组。
- 完成后自动收起成功任务。
- 保持错误和待用户操作展开。
- 补齐任务组交互埋点。

### 第三阶段：Agent 协同

- 评估是否需要 Agent 输出更稳定的任务标识和任务摘要。
- 优化复杂 run 的任务边界。
- 建立跨回合连续任务和修改反馈。

## 11. 验收标准

- 简单回答不会被包装成不必要的任务卡。
- 长任务运行中始终能看出当前正在做什么。
- 完成后无需展开过程即可找到结果、产物和下一步。
- 用户可以展开查看完整工具调用和文件修改。
- 错误、审批和问题卡不会被默认折叠。
- 同一回合只出现一个明确主动作。
- 满意度模块只在项目首次成功生成可预览产物后出现一次，失败和后续修改不触发。
- 默认收起不降低错误恢复率、审批完成率和产物打开率。

Wil je eerst de basisdefinitie, dan staat die in de primer <a href=\"/blog/what-is-vibe-design/\">what is vibe design</a>.</p>\n\n<h2>Het werkelijke verschil: hetzelfde instinct, een ander artefact</h2>\n\n<p>Beide vertrekken vanuit hetzelfde instinct — beschrijf de <em>vibe</em> (een gevoel, een richting, een referentie) en laat AI iets produceren, in plaats van elke rechthoek met de hand te plaatsen of elke div met de hand te typen. Ze splitsen op <strong>wat je overhoudt:</strong></p>\n\n<ul>\n<li><strong>Vibe design</strong> levert een <em>ontwerp</em> — een scherm, een lay-out, een mockup waar je naar kijkt en op reageert. Het artefact is een beslissing: \"ja, deze richting.\"</li>\n<li><strong>Vibe coding</strong> levert <em>code</em> — een werkende front-end die je kunt aanklikken. Het artefact is een resultaat: \"dit werkt.\"</li>\n</ul>\n\n<p>Al het andere volgt uit die ene vertakking. Een ontwerp is goedkoop te wijzigen en onmogelijk te shippen. Code ship je en is duur om opnieuw vorm te geven. Dat betekent dat de twee tools op precies tegenovergestelde manieren falen — en dát, niet de definitie, hoort je keuze te sturen.</p>\n\n<table>\n<thead>\n<tr><th></th><th>Vibe design</th><th>Vibe coding</th></tr>\n</thead>\n<tbody>\n<tr><td><strong>Je krijgt</strong></td><td>Een bewerkbaar ontwerp / mockup</td><td>Werkende front-end-code</td></tr>\n<tr><td><strong>Geoptimaliseerd voor</strong></td><td>Richting verkennen</td><td>Iets maken dat draait</td></tr>\n<tr><td><strong>Wijzigingen zijn</strong></td><td>Goedkoop — het is een plaatje</td><td>Duur — het is een build</td></tr>\n<tr><td><strong>Direct te shippen?</strong></td><td>Nee — moet nog gebouwd worden</td><td>Ja — maar gestyled zoals het model gokte</td></tr>\n<tr><td><strong>Typische tool</strong></td><td>Visily, Uizard, Figma AI</td><td>v0, Lovable, Bolt</td></tr>\n<tr><td><strong>Faalt door</strong></td><td>De mockup-klif</td><td>Design drift</td></tr>\n</tbody>\n</table>\n\n<h2>De twee faalmodi waar niemand je voor waarschuwt</h2>\n\n<p><strong>De mockup-klif (de faalmodus van vibe design).</strong> Je genereert iets prachtigs in 30 seconden, de zaal knikt instemmend, en dan blijft het daar liggen — want een mockup is een beschrijving van een app, geen app. Iemand moet het nog steeds bouwen, en hoe mooier de mockup, hoe groter de verzonken kosten als de build er anders uitkomt. Ik heb hetzelfde \"afgeronde\" scherm vaker in code herbouwd dan ik durf toe te geven. De mockup voelde als vooruitgang; het was een schuldbekentenis.</p>\n\n<p><strong>Design drift (de faalmodus van vibe coding).</strong> Dit is degene die teams verrast. Vraag een code-gen-tool om scherm na scherm en elk is <em>plausibel</em> maar subtiel verkeerd — hier een andere knopradius, daar een grijs buiten het palet, witruimte die bijna klopt. Er is een Reddit-thread die simpelweg zegt \"the design drift created by vibe coding is insane,\" en dat is het hele probleem in één regel. De code draait, dus het <em>lijkt</em> alsof het geshipt is — maar je hebt een dozijn bijna-treffers opgestapeld die samen geen systeem vormen. Niemand besloot het ontwerp; het model gokte het twaalf keer.</p>\n\n<p>En dan het deel dat ertoe doet: <strong>deze twee faalmodi hebben dezelfde grondoorzaak.</strong> De mockup-klif en design drift gebeuren allebei omdat het ontwerp en de code geen gedeelde bron van waarheid delen. De mockup weet niet hoe hij gebouwd zal worden; de code weet niet wat het ontwerpsysteem voorschrijft. Repareer dat ene ding en beide faalmodi verdwijnen grotendeels.</p>\n\n<figure>\n  <img src=\"/blog/vibe-design-od-ship.webp\" alt=\"Een warme redactionele studieplaat met een mockup en een code-artefact die samenkomen op één gedeeld design-systeembestand, getekend als verbonden engineeringdiagrammen\" />\n  <figcaption>De mockup-klif en design drift zijn dezelfde bug: ontwerp en code zonder gedeelde bron van waarheid.</figcaption>\n</figure>\n\n<h2>Welke je wanneer moet pakken</h2>\n\n<p>Sla het \"het hangt ervan af\" over. Hier is de afweging die ik zou maken:</p>\n\n<p><strong>Pak vibe design als het artefact een <em>beslissing</em> is.</strong> Je verkent richting, je hebt draagvlak van stakeholders nodig voordat iemand code schrijft, er zit nog geen engineer in de kamer, of je zit in de rommelige zero-to-one waar tien richtingen weggooien juist het doel is. Je wilt de goedkoopst mogelijke manier om opties te zien en af te wijzen. Ship het niet — beslis ermee.</p>\n\n<p><strong>Pak vibe coding als het artefact moet <em>draaien</em>.</strong> Het prototype heeft echte interactie nodig, je bent voorbij richting en zit in \"laat het werken,\" of de snelste weg naar leren is op iets levends klikken. Ga er alleen in met de wetenschap dat je hebt getekend voor het beheren van drift — wat betekent dat je een ontwerpsysteem nodig hebt dat de generatie ook echt kan volgen, niet alleen vibes.</p>\n\n<p><strong>De aanwijzing voor waar je in zit:</strong> vraag \"is de volgende beslissing <em>wat moet dit zijn</em> of <em>werkt dit</em>?\" De eerste is vibe design. De tweede is vibe coding. De meeste teams pakken de tool die ze leuk vinden in plaats van de vraag die ze eigenlijk beantwoorden — dat is de fout.</p>\n\n<h2>Het echte antwoord: maak er één loop van</h2>\n\n<p>De framing van \"design <em>vs</em> coding\" is zelf de val. Behandel je ze als losse stappen, dan krijg je een overdracht — en overdrachten zijn waar beide faalmodi leven. De teams die niet bloeden behandelen ze als één loop, en wat de loop sluit is saai: <strong>een ontwerpsysteem dat een echt, gedeeld artefact is waar beide helften zich aan houden.</strong></p>\n\n<p>Daar is <a href=\"/\">Open Design</a> op gebouwd. Het ontwerpsysteem is geen Figma-bibliotheek die de code niet kan lezen of een stijlgids die het model negeert — het is een draagbaar <code>DESIGN.md</code>-bestand dat zowel de ontwerpstap als de code-gen-stap consumeert. Genereer er een ontwerp uit, genereer er code tegenaan, en de mockup kan niet van de klif vallen (hij wijst al naar echte code) en de code kan niet afdrijven (hij is vastgepind aan het systeem). Vibe design en vibe coding houden op een tegenstelling te zijn en worden <a href=\"/blog/what-is-vibe-design/\">één beweging van intentie naar geshipt</a>. Voor de tools die vandaag aan weerszijden van de lijn leven, heb ik het veld doorgenomen in <a href=\"/blog/vibe-design-tools/\">vibe design tools</a>.</p>\n\n<h2>FAQ</h2>\n\n<p><strong>Wat is het verschil tussen vibe design en vibe coding?</strong> Vibe design genereert een ontwerp waar je op reageert; vibe coding genereert werkende code die je kunt gebruiken. Hetzelfde intentie-eerst-instinct, een ander artefact — een beslissing versus een resultaat.</p>\n\n<p><strong>Is vibe design gewoon vibe coding voor ontwerpers?</strong> Nee. Vibe coding levert code (en het ontwerp drift tenzij een systeem het inkadert); vibe design levert een ontwerp (dat nooit live gaat tenzij iemand het bouwt). Ze zijn complementaire helften, niet hetzelfde ding voor verschillende doelgroepen.</p>\n\n<p><strong>Welke moet ik eerst leren?</strong> Welke ook past bij je volgende beslissing: <em>wat moet dit zijn</em> (vibe design) of <em>werkt dit</em> (vibe coding). Op lange termijn zit de hefboom in het verbinden ervan, zodat geen van beide alleen faalt.</p>\n\n<p><strong>Vervangt vibe coding ontwerpers?</strong> Het vervangt het met de hand plaatsen van rechthoeken, niet het beslissen wat juist is. Het oordeel over <em>wat zou moeten bestaan</em> — en het systeem dat het samenhangend houdt — is precies het deel dat niet automatiseert. Drift is het bewijs.</p>\n\n<h2>De conclusie</h2>\n\n<p>Vibe design en vibe coding zijn geen concurrenten; ze zijn de twee uiteinden van één beweging, en elk team dat ze als losse stappen behandelt betaalt de tol in het midden — een mockup die nooit live gaat of een UI die afdrijft. Kies op basis van de vraag die voor je ligt: beslissen wat het moet zijn, of het laten draaien. Doe dan het ding dat bijna niemand doet — geef beide helften hetzelfde ontwerpsysteem om zich aan te houden, zodat de vibe het helemaal overleeft, van intentie tot geshipte code die je zelf bezit.</p>"

Most explanations of *vibe design vs vibe coding* stop at a definition: one makes designs, one makes code, the end. That's true and useless. I work the seam between the two at Open Design — the part where a generated design is supposed to become a running interface — and I can tell you the definition isn't where the money is. The money is in the gap *between* them: the place where a beautiful mockup never ships, and where generated code quietly drifts away from any coherent design. Get the split wrong and you pay in one of those two currencies every time.

So this isn't another glossary. It's what actually differs, the two failure modes nobody puts on the feature page, and a framework for which one to reach for when. If you want the ground-floor definition first, the [what is vibe design](/blog/what-is-vibe-design/) primer has it.

## The actual difference: same instinct, different artifact

Both start from the same instinct — describe the *vibe* (a feeling, a direction, a reference) and let AI produce something, instead of hand-placing every rectangle or hand-typing every div. They split on **what you're left holding:**

- **Vibe design** produces a *design* — a screen, a layout, a mockup you look at and react to. The artifact is a decision: "yes, this direction."
- **Vibe coding** produces *code* — a running front-end you can click. The artifact is a deliverable: "this works."

Everything else follows from that one fork. A design is cheap to change and impossible to ship. Code ships and is expensive to restyle. Which means the two tools fail in exactly opposite ways — and that, not the definition, is what should drive your choice.

| | Vibe design | Vibe coding |
|---|---|---|
| **You get** | An editable design / mockup | Running front-end code |
| **Optimized for** | Exploring direction | Producing something that runs |
| **Changes are** | Cheap — it's a picture | Expensive — it's a build |
| **Ships as-is?** | No — needs to be built | Yes — but styled however the model guessed |
| **Typical tool** | Visily, Uizard, Figma AI | v0, Lovable, Bolt |
| **Fails by** | The mockup cliff | Design drift |

## The two failure modes nobody warns you about

**The mockup cliff (vibe design's failure).** You generate something gorgeous in 30 seconds, the room nods, and then it sits there — because a mockup is a description of an app, not an app. Someone still has to build it, and the prettier the mockup, the more sunk cost when the build comes in different. I've rebuilt the same "finished" screen in code more times than I'd like to admit. The mockup felt like progress; it was a promissory note.

**Design drift (vibe coding's failure).** This is the one that surprises teams. Ask a code-gen tool for screen after screen and each one is *plausible* but subtly off — a different button radius here, an off-palette gray there, spacing that almost matches. There's a Reddit thread that just says "the design drift created by vibe coding is insane," and that's the whole problem in one line. The code runs, so it *looks* like it shipped — but you've accumulated a dozen near-misses that don't add up to a system. Nobody decided the design; the model guessed it twelve times.

Here's the part that matters: **these two failures have the same root cause.** The mockup cliff and design drift both happen because the design and the code don't share a source of truth. The mockup doesn't know how it'll be built; the code doesn't know what the design system says. Fix that one thing and both failure modes mostly disappear.

## Which to reach for, when

Skip the "it depends." Here's the call I'd make:

**Reach for vibe design when the artifact is a *decision*.** You're exploring direction, you need stakeholder buy-in before anyone writes code, there's no engineer in the room yet, or you're at the messy zero-to-one where throwing away ten directions is the point. You want the cheapest possible way to see and reject options. Don't ship it — decide with it.

**Reach for vibe coding when the artifact has to *run*.** The prototype needs real interaction, you're past direction and into "make it work," or the fastest path to learning is clicking a live thing. Just go in knowing you've signed up to manage drift — which means you need a design system the generation can actually follow, not vibes alone.

**The tell for which you're in:** ask "is the next decision *what should this be* or *does this work*?" The first is vibe design. The second is vibe coding. Most teams reach for the tool they like instead of the question they're actually answering — that's the mistake.

## The real answer: make them one loop

The framing of "design *vs* coding" is itself the trap. Treated as separate steps, you get a handoff — and handoffs are where both failure modes live. The teams that don't bleed treat them as one loop, and the thing that closes the loop is boring: **a design system that's a real, shared artifact both halves obey.**

That's the bet [Open Design](/) is built on. The design system isn't a Figma library the code can't read or a style guide the model ignores — it's a portable `DESIGN.md` file that both the design step and the code-gen step consume. Generate a design from it, generate code against it, and the mockup can't cliff (it's already pointed at real code) and the code can't drift (it's pinned to the system). Vibe design and vibe coding stop being a versus and become [one motion from intent to shipped](/blog/what-is-vibe-design/). For the tools that live on each side of the line today, I went through the field in [vibe design tools](/blog/vibe-design-tools/).

## FAQ

**What's the difference between vibe design and vibe coding?** Vibe design generates a design you react to; vibe coding generates running code you can use. Same intent-first instinct, different artifact — a decision versus a deliverable.

**Is vibe design just vibe coding for designers?** No. Vibe coding produces code (and the design drifts unless a system constrains it); vibe design produces a design (that never ships unless someone builds it). They're complementary halves, not the same thing for different audiences.

**Which should I learn first?** Whichever matches your next decision: *what should this be* (vibe design) or *does this work* (vibe coding). Long term, the leverage is in connecting them so neither fails alone.

**Does vibe coding replace designers?** It replaces hand-placing rectangles, not deciding what's right. The judgment about *what should exist* — and the system that keeps it coherent — is exactly the part that doesn't automate. Drift is the proof.

## The takeaway

Vibe design and vibe coding aren't competitors; they're the two ends of one motion, and every team that treats them as separate steps pays the toll in the middle — a mockup that never ships or a UI that drifts. Pick by the question in front of you: deciding what it should be, or making it run. Then do the thing almost nobody does — give both halves the same design system to obey, so the vibe survives all the way from intent to shipped code you own.

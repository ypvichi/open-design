      <p>直接让 Claude Code「做个落地页」，你往往会得到和所有人一样的平庸结果——保险的字体、默认的蓝色、毫无主张。这并不是模型能力的天花板，而是提示词和配置的问题。配上合适的插件，再养成几个习惯，Claude Code 做出来的前端设计就能真正有品味。这篇是实操版：怎么配置、怎么提示，以及如何把产出从一个好看的单屏升级成一套你能真正交付并拥有的设计系统。</p>
      <p>我在 Open Design 负责「设计到代码」这条链路，每天都拿 Claude Code 跑真实的设计需求。我们就在这个领域里做产品，所以我有自己的立场——我也会坦白说明官方工具到哪儿为止、像我们这样的东西又从哪儿接上。但这篇指南的绝大部分，纯粹就是怎么把 Claude Code 的前端设计能力榨干。</p>
      <h2>第一步：安装 frontend-design 插件</h2>
      <p>Anthropic 为 Claude Code 提供了官方的 <a href="https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design">frontend-design 插件</a>，它是对设计质量提升最大的一招。在 Claude Code 里：</p>
      <ol>
      <li>输入 <code>/plugin</code>。</li>
      <li>选择 <strong>Add Marketplace</strong>，输入 <code>anthropics/claude-code</code>。</li>
      <li>找到 <strong>frontend-design</strong> 并安装。</li>
      </ol>
      <p>装好之后，只要你让 Claude 构建界面，这个 skill 就会自动激活。它的作用是推开默认值：在写任何代码之前，先确立一套设计框架——目的、受众、一个明确的审美方向——这样你拿到的是有辨识度的排版、刻意的配色和经过推敲的动效，而不是模板化的产出。</p>
      <h2>第二步：用审美方向提示，而不是像素值</h2>
      <p>最大的错误就是过度指定。别把一堆边距和十六进制色值的规格丢给 Claude；给它一个<em>方向</em>，让它在这个框架内自己做选择。告诉它该考虑什么：</p>
      <ul>
      <li><strong>目的与受众</strong>——「一个面向开发者工具的落地页，要有精准、迅捷的感觉」，而不是「做个落地页」。</li>
      <li><strong>调性</strong>——冷静、编辑感，或大胆、高对比，又或者复古终端风。</li>
      <li><strong>字体类别</strong>——「正文用人文主义无衬线，标题用有辨识度的展示字体」，胜过点名某一款具体字体。</li>
      <li><strong>色彩族系</strong>——「暖中性色配一抹荧光强调色」给它留了余地；「#63fe13 的按钮」则没有。</li>
      <li><strong>动效理念</strong>——「克制、退场迅速」对比「俏皮、有弹性」。</li>
      </ul>
      <p>审美方向就是把 <a href="/blog/what-is-vibe-design/">vibe design</a> 的思路用在 Claude Code 上：你描述感觉和约束，由 agent 来填补手艺。</p>
      <h2>第三步：一次只引导一个设计维度</h2>
      <p>当第一版已经接近但还显得平庸时，别推倒重来——把 Claude 的注意力一次集中到一个维度上。下面每一项都是你可以独立拉动的杠杆：</p>
      <table><thead><tr><th>维度</th><th>弱提示</th><th>强提示</th></tr></thead><tbody><tr><td><strong>排版</strong></td><td>「字体好看点」</td><td>「字号对比更强——超大号展示标题、小型大写字母的标签」</td></tr><tr><td><strong>色彩</strong></td><td>「换个颜色」</td><td>「降到接近单色的基底，只留一个高饱和强调色」</td></tr><tr><td><strong>动效</strong></td><td>「加点动画」</td><td>「入场淡入约 200ms，退场干脆约 140ms，不要回弹」</td></tr><tr><td><strong>背景</strong></td><td>「别那么素」</td><td>「淡淡的点阵网格纹理，不要渐变」</td></tr><tr><td><strong>参照</strong></td><td>「做得现代点」</td><td>「往 Linear／IDE 深色主题那种审美靠」</td></tr></tbody></table>
      <p>点名一个参照（某个 IDE 主题、某个品牌、某种文化审美）是把 Claude 从默认值里拽出来最快的办法——它给了模型一个具体的目标，而不是一个平均值。</p>
      <h2>第四步：分层下达需求，并为多轮迭代做规划</h2>
      <p>把第一版当成地基，而不是成品功能。两个会越用越省力的习惯：</p>
      <ul>
      <li><strong>分层构建：</strong>先类型，再逻辑，然后 UI，最后测试。一条提示词里要求所有东西只会得到一团乱麻；分层能让每一遍都可审查。</li>
      <li><strong>规划 3–5 轮迭代。</strong>第一屏确立方向；第 2 到第 5 遍才是品味出现的地方。每一轮都对照你的审美方向来评审，而不是逐像素地抠。</li>
      </ul>
      <p>如果你的原型需要真正能跑起来，而不只是看着对，那就是 <a href="/blog/design-to-code-tools/">vibe design 与 vibe coding</a> 的分界线——Claude Code 在两边都强，因为设计从一开始<em>就是</em>代码。</p>
      <h2>第五步：从一次性单屏到一套可拥有的设计系统</h2>
      <p>到这里，官方插件的职责就结束了，更难的问题才刚开始——我也会坦诚地谈谈我们自己的工具。frontend-design 插件能让<em>单个屏幕</em>看起来很棒。但一个产品是四十个必须保持连贯的屏幕，是一套要在各个功能间存活下去的设计系统，是你要维护一年的代码。逐屏独立提示，你得到的是四十个有品味却互不一致的页面，以及一套只活在你提示历史里的设计系统。</p>
      <p>解法是把设计系统变成 Claude Code 能读取的东西，而不是每次提示都重新描述一遍。这正是 <a href="/">Open Design</a> 在 Claude Code 之上补的东西：每套设计系统都成为一个 <code>DESIGN.md</code>，每项可复用能力都成为一个 <code>SKILL.md</code>——都是你的 agent 会加载的纯文件，于是「做设置页」会继承和其他一切相同的字号梯度、色彩系统与组件，产出也就<a href="/blog/what-is-vibe-design/">从提示词走到你拥有的、可交付的代码</a>。<em>诚实地划清边界：</em>对于单个页面或一个快速原型，光用插件就完全够了——当一个真实产品的跨页面一致性、以及对这些文件的所有权开始变得重要时，再上设计系统这一层。看看它如何契合<a href="/solutions/designer/">设计师</a>和<a href="/solutions/engineering/">工程团队</a>。</p>
      <h2>常见错误</h2>
      <ul>
      <li><strong>过度指定像素。</strong>你会把模型压扁成一个渲染器。给方向，让它来选。</li>
      <li><strong>一条超级提示词包办一切。</strong>改成分层：类型 → 逻辑 → UI → 测试。</li>
      <li><strong>指望一次就完美。</strong>预留 3–5 轮迭代；对照 vibe 评审，而不是像素。</li>
      <li><strong>多屏工作没有设计系统。</strong>逐屏提示会漂移；把系统放进 agent 能读取的文件里。</li>
      <li><strong>接受第一套配色／字体。</strong>默认值就是平均值；点名一个参照来逃离它们。</li>
      </ul>
      <h2>常见问题</h2>
      <p><strong>Claude Code 真能做出好的前端设计吗？</strong>能，配上 frontend-design 插件和以方向为主导的提示方式。没有它们你拿到的是平庸的默认值；有了它们你拿到的是有辨识度、有意图的 UI。</p>
      <p><strong>怎么安装 Claude Code 的 frontend-design 插件？</strong>在 Claude Code 里输入 <code>/plugin</code> → Add Marketplace → <code>anthropics/claude-code</code> → 安装 <strong>frontend-design</strong>。之后只要你请求构建界面，它就会自动激活。</p>
      <p><strong>该怎么给 Claude Code 写设计提示？</strong>用审美方向（目的、调性、字体类别、色彩族系、动效理念）和参照，而不是像素值——然后迭代 3–5 次，一次引导一个维度。</p>
      <p><strong>怎么让设计在很多屏幕之间保持一致？</strong>把设计系统从提示词里挪出来，放进 agent 能读取的文件里。像 <a href="/">Open Design</a> 这样的 agent 原生层会把每套设计系统变成 Claude Code 每次构建都加载的 <code>DESIGN.md</code>。想了解它在更大的格局里处于什么位置，看看 <a href="/blog/best-ai-design-tools/">最佳 AI 设计工具</a>指南。</p>
      <p><strong>Claude Code 比专门的 AI 设计工具更好吗？</strong>形态不同：Claude Code <em>以代码方式</em>做设计，所以没有从设计稿到代码的交接环节——权衡之处见 <a href="/blog/design-to-code-tools/">设计到代码工具</a>的对比。</p>
      <h2>要点回顾</h2>
      <p>Claude Code 的前端设计水平，取决于你的配置和提示：安装 <a href="https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design">frontend-design 插件</a>，用审美方向而非像素来提示，一次引导一个设计维度，并做好迭代的打算。这能让你得到真正出色的单屏。而要让整个产品保持连贯、并真正拥有成果，就把设计系统放进你的 agent 能读取的文件里——这正是 <a href="/">Open Design</a> 押注的方向：你的 agent，你的、以 <code>DESIGN.md</code> 形式存在的设计系统，从提示词直达可交付。</p>

      <p>私は <strong>Open Design</strong> でデザインからコードへのパイプラインに取り組んでおり、日々、実際のデザインブリーフに対して Claude Code を走らせています。私たちはこの領域でプロダクトを作っているので利害関係があります。だからこそ、公式ツールでカバーできる範囲がどこまでで、私たちのようなツールの出番がどこから始まるのかは率直にお伝えします。とはいえ本ガイドの大部分は、純粋に Claude Code から優れたフロントエンドデザインを引き出す方法そのものです。</p>
      <h2>ステップ1：frontend-design プラグインをインストールする</h2>
      <p>Anthropic は Claude Code 向けに公式の <a href="https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design">frontend-design プラグイン</a>を提供しており、これがデザイン品質を最も大きく引き上げる単一の要素です。Claude Code 内で次のように操作します。</p>
      <ul><li><code>/plugin</code> と入力します。</li>
      <li><strong>Add Marketplace</strong> を選び、<code>anthropics/claude-code</code> を入力します。</li>
      <li><strong>frontend-design</strong> を探してインストールします。</li></ul>
      <p>インストールすると、インターフェースの構築を依頼するたびにこのスキルが自動的に有効になります。その役割はデフォルトを超えていくことです。コードを書く前にまずデザインフレームワーク（目的、対象ユーザー、具体的な美的方向性）を確立するため、テンプレートのような出力ではなく、特徴的なタイポグラフィ、意図のあるカラー、練られたモーションが得られます。</p>
      <h2>ステップ2：ピクセル値ではなく美的方向性でプロンプトを書く</h2>
      <p>最大の間違いは指定しすぎることです。余白や16進カラーコードの仕様書を Claude に渡してはいけません。渡すのは<em>方向性</em>であり、その枠の中で選択はモデルに委ねます。何を考えるべきかを伝えましょう。</p>
      <ul><li><strong>目的と対象ユーザー</strong>——「ランディングページを作って」ではなく、「精密さとスピードを感じさせるべき開発者向けツールのランディングページ」。</li>
      <li><strong>トーン</strong>——落ち着いた編集的な雰囲気か、大胆でハイコントラストか、あるいはレトロなターミナル風か。</li>
      <li><strong>フォントのカテゴリ</strong>——特定のフォント名を挙げるより、「本文はヒューマニスト系サンセリフ、見出しは個性的なディスプレイ書体」の方が効果的です。</li>
      <li><strong>カラーファミリー</strong>——「温かみのあるニュートラルに一点だけ鮮烈なアクセント」なら余地が生まれます。「#63fe13 のボタン」では生まれません。</li>
      <li><strong>モーションの思想</strong>——「抑制的で、退場は素早く」対「遊び心があって弾むように」。</li></ul>
      <p>美的方向性とは、<a href="/blog/what-is-vibe-design/">バイブデザイン</a>のアプローチを Claude Code に適用したものです。あなたが感覚と制約を描写し、エージェントが作り込みを埋めていきます。</p>
      <h2>ステップ3：デザインの各次元を一つずつ導く</h2>
      <p>最初の結果が惜しいけれどありきたりなときは、やり直すのではなく、Claude の注意を一度に一つの次元へ向けさせます。以下のそれぞれが、独立して操作できるレバーです。</p>
      <table><thead><tr><th>次元</th><th>弱いプロンプト</th><th>強いプロンプト</th></tr></thead><tbody><tr><td><strong>タイポグラフィ</strong></td><td>「もっと良いフォントに」</td><td>「文字のコントラストをもっと——特大のディスプレイ見出し、スモールキャップスのラベル」</td></tr><tr><td><strong>カラー</strong></td><td>「色を変えて」</td><td>「ほぼモノクロのベースに、一点だけ彩度の高いアクセントへ落とす」</td></tr><tr><td><strong>モーション</strong></td><td>「アニメーションを足して」</td><td>「登場のフェードは約200ms、退場は約140msで決然と、バウンスなし」</td></tr><tr><td><strong>背景</strong></td><td>「もっと地味じゃなく」</td><td>「控えめなドットグリッドの質感、グラデーションなし」</td></tr><tr><td><strong>参照</strong></td><td>「モダンにして」</td><td>「Linear や IDE のダークテーマの美学に寄せて」</td></tr></tbody></table>
      <p>参照対象（IDE のテーマ、ブランド、文化的な美学）を名指しするのは、Claude をデフォルトから抜け出させる最速の方法です。平均値ではなく具体的なターゲットをモデルに与えられるからです。</p>
      <h2>ステップ4：依頼を階層化し、反復を前提に計画する</h2>
      <p>最初のバージョンは完成した機能ではなく土台として扱いましょう。積み重なる効果のある二つの習慣があります。</p>
      <ul><li><strong>ビルドを階層化する：</strong>まず型、次にロジック、次に UI、最後にテスト。すべてを一つのプロンプトで求めると絡まった塊になりますが、階層化すれば各段階をレビュー可能に保てます。</li>
      <li><strong>3〜5回の反復を計画する。</strong>最初の画面は方向性を確立するもので、2〜5回目こそセンスが宿る段階です。毎回、ピクセル単位ではなくあなたの美的方向性に照らしてレビューします。</li></ul>
      <p>プロトタイプが見た目だけでなく実際に動く必要があるなら、それが<a href="/blog/design-to-code-tools/">バイブデザインとバイブコーディング</a>の境界です。Claude Code は最初からデザイン<em>そのもの</em>がコードなので、どちらにも強いのです。</p>
      <h2>ステップ5：一回限りの画面から、自分のものにできるデザインシステムへ</h2>
      <p>ここが公式プラグインの役割が終わり、より難しい問題が始まる地点であり、私たちのツールについて正直にお話しする場面です。frontend-design プラグインは<em>単一の画面</em>を見事に仕上げます。しかしプロダクトとは、一貫性を保たねばならない40個の画面であり、複数の機能をまたいで生き延びるべきデザインシステムであり、1年間メンテナンスし続けるコードです。各画面を個別にプロンプトすれば、センスはあるけれどバラバラな40ページと、プロンプト履歴の中にしか存在しないデザインシステムが出来上がります。</p>
      <p>解決策は、デザインシステムを毎回プロンプトで説明し直すものではなく、Claude Code が読み込む対象にすることです。それこそ <a href="/">Open Design</a> が Claude Code の上に付け加えるものです。すべてのデザインシステムは <code>DESIGN.md</code> となり、再利用可能なすべての機能は <code>SKILL.md</code> になります。これらはエージェントが読み込むただのプレーンなファイルです。だから「設定ページを作って」は、ほかのすべてと同じタイプスケール、カラーシステム、コンポーネントを継承し、出力は<a href="/blog/what-is-vibe-design/">プロンプトから、自分が所有する出荷可能なコードへ</a>と至ります。<em>正直な適用範囲：</em>単一ページや手早いプロトタイプなら、プラグインだけで十分です。デザインシステムのレイヤーに手を伸ばすのは、実プロダクト全体での一貫性とファイルの所有権が重要になり始めたときです。<a href="/solutions/designer/">デザイナー</a>や<a href="/solutions/engineering/">エンジニアリングチーム</a>への適合のしかたもご覧ください。</p>
      <h2>よくある間違い</h2>
      <ul><li><strong>ピクセルを指定しすぎる。</strong>モデルを単なるレンダラーへと平板化してしまいます。方向性を与え、選ばせましょう。</li>
      <li><strong>一つの巨大プロンプト。</strong>型→ロジック→UI→テスト、と階層化しましょう。</li>
      <li><strong>一発で完璧を期待する。</strong>3〜5回の反復を見込み、ピクセルではなくバイブに照らしてレビューします。</li>
      <li><strong>複数画面の作業にデザインシステムがない。</strong>画面ごとのプロンプトはぶれていきます。システムはエージェントが読むファイルに置きましょう。</li>
      <li><strong>最初のパレットやフォントを受け入れる。</strong>デフォルトは平均値です。そこから抜け出すには参照を名指ししましょう。</li></ul>
      <h2>FAQ</h2>
      <p><strong>Claude Code は本当に質の高いフロントエンドデザインができますか？</strong>はい、frontend-design プラグインと方向性主導のプロンプトがあればできます。それらがなければありきたりなデフォルトになりますが、それらがあれば特徴的で意図のある UI が得られます。</p>
      <p><strong>Claude Code の frontend-design プラグインはどうインストールしますか？</strong>Claude Code で <code>/plugin</code> と入力 → Add Marketplace → <code>anthropics/claude-code</code> → <strong>frontend-design</strong> をインストール。あとはインターフェースを依頼すると自動的に有効になります。</p>
      <p><strong>Claude Code にデザインのプロンプトをどう書けばいいですか？</strong>ピクセル値ではなく、美的方向性（目的、トーン、フォントのカテゴリ、カラーファミリー、モーションの思想）と参照対象で。そのうえで、一度に一つの次元を導きながら3〜5回反復します。</p>
      <p><strong>多数の画面でデザインの一貫性をどう保てばいいですか？</strong>デザインシステムをプロンプトの外へ出し、エージェントが読むファイルに移します。<a href="/">Open Design</a> のようなエージェントネイティブのレイヤーは、各デザインシステムを Claude Code がビルドのたびに読み込む <code>DESIGN.md</code> に変えます。より広い全体像の中での位置づけは<a href="/blog/best-ai-design-tools/">最良の AI デザインツール</a>のガイドをご覧ください。</p>
      <p><strong>Claude Code は専用の AI デザインツールより優れていますか？</strong>形が違います。Claude Code は<em>コードとして</em>デザインするため、モックアップからコードへの引き継ぎがありません。トレードオフについては<a href="/blog/design-to-code-tools/">デザインからコードへのツール</a>の比較をご覧ください。</p>
      <h2>まとめ</h2>
      <p>Claude Code のフロントエンドデザインは、あなたのセットアップとプロンプト次第で良くも悪くもなります。<a href="https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design">frontend-design プラグイン</a>をインストールし、ピクセルではなく美的方向性でプロンプトを書き、デザインの次元を一度に一つずつ導き、反復する計画を立てましょう。これで本当に質の高い単発の画面が得られます。プロダクト全体の一貫性を保ち、その成果を自分のものにするには、デザインシステムをエージェントが読むファイルに置くこと——それこそ <a href="/">Open Design</a> が賭けているものです。あなたのエージェント、<code>DESIGN.md</code> としてのあなたのデザインシステム、プロンプトから出荷へ。</p>

      <h2>1단계: frontend-design 플러그인 설치하기</h2>
      <p>Anthropic은 Claude Code용 공식 <a href="https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design">frontend-design 플러그인</a>을 제공하는데, 이것이 디자인 품질을 끌어올리는 가장 큰 한 방입니다. Claude Code에서:</p>
      <ol><li><code>/plugin</code>을 입력합니다.</li><li><strong>Add Marketplace</strong>를 선택하고 <code>anthropics/claude-code</code>를 입력합니다.</li><li><strong>frontend-design</strong>으로 이동해 설치합니다.</li></ol>
      <p>설치하고 나면, 인터페이스를 만들어 달라고 요청할 때마다 이 스킬이 자동으로 활성화됩니다. 이 스킬의 역할은 기본값을 넘어서도록 밀어붙이는 것입니다. 코드를 한 줄이라도 쓰기 전에 디자인 프레임워크—목적, 대상, 구체적인 미감의 방향성—를 먼저 세우기 때문에, 틀에 박힌 산출물 대신 개성 있는 타이포그래피, 의도가 담긴 색, 신중하게 다룬 모션을 얻게 됩니다.</p>
      <h2>2단계: 픽셀 값이 아니라 미감의 방향성으로 프롬프트하기</h2>
      <p>가장 흔한 실수는 지나치게 구체적으로 지정하는 것입니다. Claude에게 여백과 헥스 코드로 가득한 명세를 넘기지 마세요. <em>방향성</em>을 건네고, 그 틀 안에서 선택은 모델이 하도록 두세요. 무엇을 고민해야 할지를 알려주세요:</p>
      <ul><li><strong>목적과 대상</strong> — "랜딩 페이지를 만들어줘"가 아니라 "정밀하고 빠르다는 인상을 줘야 하는 개발자 도구 랜딩 페이지".</li><li><strong>톤</strong> — 차분하고 에디토리얼한 느낌, 혹은 과감하고 대비가 강한 느낌, 혹은 레트로 터미널풍.</li><li><strong>폰트 카테고리</strong> — 특정 폰트 이름을 지정하기보다 "본문은 휴머니스트 산세리프, 제목은 개성 있는 디스플레이 서체"가 낫습니다.</li><li><strong>색 계열</strong> — "따뜻한 뉴트럴에 강렬한 애시드 톤 포인트 하나"는 여지를 주지만, "#63fe13 버튼"은 그렇지 않습니다.</li><li><strong>모션 철학</strong> — "절제되고 빠른 퇴장" 대 "장난스럽고 통통 튀는".</li></ul>

      <h2>4단계: 요청을 겹겹이 쌓고 반복을 염두에 두기</h2>
      <p>첫 버전은 완성된 기능이 아니라 토대로 다루세요. 효과가 누적되는 두 가지 습관:</p>
      <ul><li><strong>빌드를 층층이 쌓기:</strong> 타입을 먼저, 그다음 로직, 그다음 UI, 마지막에 테스트. 한 프롬프트에 모든 걸 요청하면 뒤엉킨 결과가 나옵니다. 층을 나누면 각 단계를 검토하기 쉬워집니다.</li><li><strong>3~5회 반복을 계획하기.</strong> 첫 화면은 방향을 잡는 단계이고, 안목이 발휘되는 곳은 2~5회 차입니다. 매 회차마다 픽셀 단위가 아니라 미감의 방향성을 기준으로 검토하세요.</li></ul>
      <p>프로토타입이 보기에 그럴듯한 데 그치지 않고 실제로 동작해야 한다면, 그것이 바로 <a href="/blog/design-to-code-tools/">바이브 디자인 대 바이브 코딩</a>의 경계입니다. Claude Code는 처음부터 디자인 <em>자체가</em> 코드이기 때문에 양쪽 모두에 강합니다.</p>
      <h2>5단계: 일회성 화면에서 소유 가능한 디자인 시스템으로</h2>
      <p>바로 여기서 공식 플러그인의 역할이 끝나고 더 어려운 문제가 시작됩니다. 그리고 여기서 우리 자신의 도구에 대해 투명하게 말하겠습니다. frontend-design 플러그인은 <em>단일 화면</em>을 멋지게 만들어 줍니다. 하지만 제품이란 일관성을 유지해야 하는 마흔 개의 화면이고, 여러 기능을 거쳐 살아남아야 하는 디자인 시스템이며, 1년 동안 유지보수할 코드입니다. 각 화면을 따로따로 프롬프트하면, 보기엔 괜찮지만 서로 어긋나는 마흔 개의 페이지와, 오직 프롬프트 기록 속에만 존재하는 디자인 시스템을 얻게 됩니다.</p>
      <p>해법은 디자인 시스템을, 프롬프트마다 다시 설명하는 무언가가 아니라 Claude Code가 읽어 들이는 대상으로 만드는 것입니다. 그것이 <a href="/">Open Design</a>이 Claude Code 위에 더해주는 것입니다. 모든 디자인 시스템은 <code>DESIGN.md</code>가 되고, 재사용 가능한 모든 역량은 <code>SKILL.md</code>가 됩니다. 에이전트가 불러오는 평범한 파일들이죠. 그래서 "설정 페이지를 만들어줘"라고 하면 다른 모든 것과 동일한 타입 스케일, 색 시스템, 컴포넌트를 그대로 물려받고, 결과물은 <a href="/blog/what-is-vibe-design/">프롬프트에서 출시 코드까지</a> 여러분이 소유하는 형태로 나옵니다. <em>솔직한 적용 범위:</em> 단일 페이지나 빠른 프로토타입이라면 플러그인만으로도 충분합니다. 실제 제품 전반의 일관성과 파일에 대한 소유가 중요해지기 시작할 때 디자인 시스템 계층을 도입하세요. <a href="/solutions/designer/">디자이너</a>와 <a href="/solutions/engineering/">엔지니어링 팀</a>에 어떻게 맞물리는지도 확인해 보세요.</p>
      <h2>흔한 실수</h2>
      <ul><li><strong>픽셀을 지나치게 지정하기.</strong> 모델을 한낱 렌더러로 납작하게 만드는 셈입니다. 방향을 주고, 선택은 맡기세요.</li><li><strong>거대한 단일 프롬프트.</strong> 대신 타입 → 로직 → UI → 테스트 순으로 층을 나누세요.</li><li><strong>단번에 완벽하길 기대하기.</strong> 3~5회 반복을 예산으로 잡고, 픽셀이 아니라 바이브를 기준으로 검토하세요.</li><li><strong>여러 화면 작업에 디자인 시스템이 없음.</strong> 화면별 프롬프트는 점점 어긋납니다. 시스템을 에이전트가 읽는 파일에 담으세요.</li><li><strong>첫 팔레트/폰트를 그냥 받아들이기.</strong> 기본값은 곧 평균입니다. 거기서 벗어나려면 레퍼런스를 이름으로 짚어주세요.</li></ul>
      <h2>자주 묻는 질문</h2>
      <p><strong>Claude Code가 정말로 좋은 프런트엔드 디자인을 할 수 있나요?</strong> 네, frontend-design 플러그인과 방향성 중심의 프롬프트가 있으면 됩니다. 그게 없으면 뻔한 기본값이 나오고, 있으면 개성 있고 의도가 담긴 UI를 얻습니다.</p>
      <p><strong>Claude Code의 frontend-design 플러그인은 어떻게 설치하나요?</strong> Claude Code에서 <code>/plugin</code> 입력 → Add Marketplace → <code>anthropics/claude-code</code> → <strong>frontend-design</strong> 설치. 그러면 인터페이스를 요청할 때 자동으로 활성화됩니다.</p>

Out of the box, ask Claude Code to "build a landing page" and you'll often get the same generic result everyone else gets — safe fonts, default blue, no point of view. That's not a ceiling on the model; it's a prompting and setup problem. With the right plugin and a few habits, Claude Code does frontend design with real taste. This guide is the practical version: how to set it up, how to prompt it, and how to take the output from a nice one-off screen to a design system you can actually ship and own.

I work on the design-to-code pipeline at Open Design, so I run Claude Code against real design briefs daily. We build in this space, so I have a stake — and I'll be clear about where the official tooling ends and where something like ours starts. Most of this guide is just how to get great frontend design out of Claude Code, full stop.

## Step 1: Install the frontend-design plugin

Anthropic ships an official [frontend-design plugin](https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design) for Claude Code, and it's the single biggest upgrade to design quality. In Claude Code:

1. Type `/plugin`.
2. Choose **Add Marketplace** and enter `anthropics/claude-code`.
3. Browse to **frontend-design** and install it.

Once installed, the skill activates automatically whenever you ask Claude to build an interface. Its job is to push past defaults: it establishes a design framework first — purpose, audience, a specific aesthetic direction — before writing any code, so you get distinctive typography, deliberate color, and considered motion instead of template output.

## Step 2: Prompt with aesthetic direction, not pixel values

The biggest mistake is over-specifying. Don't hand Claude a spec of margins and hex codes; hand it a *direction* and let it make the choices inside that frame. Tell it what to think about:

- **Purpose and audience** — "a developer tool landing page that should feel precise and fast," not "make a landing page."
- **Tone** — calm and editorial, or bold and high-contrast, or retro-terminal.
- **Font category** — "a humanist sans for body, a distinctive display face for headings" beats naming a specific font.
- **Color family** — "warm neutrals with one acid accent" gives it room; "#63fe13 buttons" doesn't.
- **Motion philosophy** — "restrained, fast exits" vs "playful, springy."

Aesthetic direction is the [vibe design](/blog/what-is-vibe-design/) approach applied to Claude Code: you describe the feeling and the constraints, the agent fills in the craft.

## Step 3: Guide the design dimensions one at a time

When the first result is close but generic, don't restart — direct Claude's attention to one dimension at a time. Each of these is a lever you can pull independently:

| Dimension | Weak prompt | Strong prompt |
|---|---|---|
| **Typography** | "nicer fonts" | "more type contrast — oversized display headings, small-caps labels" |
| **Color** | "different colors" | "drop to a near-monochrome base with a single saturated accent" |
| **Motion** | "add animation" | "entrance fades ~200ms, decisive exits ~140ms, no bounce" |
| **Background** | "less plain" | "subtle dot-grid texture, no gradients" |
| **Reference** | "make it modern" | "lean toward a Linear/IDE-dark-theme aesthetic" |

Naming a reference (an IDE theme, a brand, a cultural aesthetic) is the fastest way to break Claude out of defaults — it gives the model a concrete target instead of an average.

## Step 4: Layer requests and plan for iterations

Treat the first version as a foundation, not a finished feature. Two habits that compound:

- **Layer the build:** types first, then logic, then UI, then tests. Asking for everything in one prompt produces a tangle; layering keeps each pass reviewable.
- **Plan 3–5 iterations.** The first screen establishes the direction; passes 2–5 are where taste happens. Review against your aesthetic direction each round, not pixel by pixel.

If your prototype needs to actually run rather than just look right, that's the [vibe design vs vibe coding](/blog/design-to-code-tools/) line — Claude Code is strong on both because the design *is* code from the start.

## Step 5: From a one-off screen to an ownable design system

Here's where the official plugin's job ends and a harder problem begins — and where I'll be transparent about our own tool. The frontend-design plugin makes a *single screen* look great. But a product is forty screens that must stay coherent, a design system that has to survive across features, and code you'll maintain for a year. Prompt each screen independently and you get forty tasteful-but-inconsistent pages and a design system that lives only in your prompt history.

The fix is to make the design system a thing Claude Code reads, not something you re-describe every prompt. That's what [Open Design](/) adds on top of Claude Code: every design system becomes a `DESIGN.md` and every reusable capability a `SKILL.md` — plain files your agent loads, so "build the settings page" inherits the same type scale, color system, and components as everything else, and the output goes [from prompt to shipped code](/blog/what-is-vibe-design/) you own. *Honest scope:* for a single page or a quick prototype, the plugin alone is plenty — reach for a design-system layer when consistency across a real product and ownership of the files start to matter. See how it fits [designers](/solutions/designer/) and [engineering teams](/solutions/engineering/).

## Common mistakes

- **Over-specifying pixels.** You flatten the model into a renderer. Give direction; let it choose.
- **One mega-prompt.** Layer types → logic → UI → tests instead.
- **Expecting one-shot perfection.** Budget 3–5 iterations; review against the vibe, not the pixels.
- **No design system for multi-screen work.** Per-screen prompting drifts; put the system in files the agent reads.
- **Accepting the first palette/font.** The defaults are the average; name a reference to escape them.

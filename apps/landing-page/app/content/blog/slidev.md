      <p>如果你会写代码，又不太想打开 PowerPoint，那你多半已经碰到过 <strong>Slidev</strong>。它自称"面向开发者的演示幻灯片"，事实也正是如此：你用 Markdown 撰写每一页幻灯片，在需要的地方嵌入 Vue 组件和代码块，然后把整套幻灯片当作一个网页来演示——热重载、语法高亮、演讲者备注、交互式嵌入内容一应俱全。凭借 GitHub 上 47.6k 颗星和 MIT 许可证，当有人问"怎么用 Markdown 做出好看的幻灯片"时，它是首选答案之一。</p>
      <p>这份指南会诚实地为它画一张地图。Slidev 究竟是什么、主题和示例如何配合、导出 PDF 和 PowerPoint 的真实表现（这里有个坑）、它与其他开源竞争者相比如何，以及——大多数文章都跳过的部分——像这样一个工具什么时候不再是对的选择、又该由别的东西接手。如果你正在为一套 <a href="/blog/claude-ppt-skills/">AI 与 Agent 辅助的幻灯片工作流</a>权衡各种选项，先弄清楚 Slidev 是什么、不是什么，会很有帮助。</p>
      <h2>Slidev 是什么</h2>
      <p>Slidev 运行在浏览器里，把你的幻灯片当作一个项目，而不是一份文档。你在单个 Markdown 文件里撰写幻灯片，用 <code>---</code> 分隔各页，而开发者工具链正是它的核心：边打字边热重载、带行聚焦的一流代码高亮、LaTeX 和图表、带备注的演讲者模式，以及一个你从终端启动的开发服务器。当纯 Markdown 走到尽头时，你可以把 <strong>Vue 组件</strong>直接嵌进某一页——于是图表、实时计数器，或你自己写的交互小部件，就能内联地摆在要点旁边。因为输出本身<em>就是</em>一个 Web 应用，Web 能做的任何事都能出现在幻灯片上。</p>
      <p>取舍就明明白白写在它的定位里：这是一个 <strong>开发者工具，而不是图形界面</strong>。你的工作流是代码编辑器加开发服务器，而不是带拖拽手柄的画布——这正是已经喜欢 Markdown 和 Web 组件的开发者会选它的原因，也正是非程序员会被它劝退的原因。它也是货真价实的免费：不用注册、没有水印、没有导出上限、没有按席位收费。Markdown 和围绕它的整个项目都归你所有。</p>
      <h2>主题与示例</h2>
      <p>围绕 Slidev 的大多数搜索，其实都是想让它<em>看起来</em>好看，又不用从零设计每一页，所以下面讲讲这些部分是怎么拼起来的。</p>
      <p><strong>Slidev 主题</strong>是内置的起点，也是它真正的一大优势。你在 frontmatter 里用一行设置主题（例如 <code>theme: seriph</code>），Slidev 就会把它拉进来。官方有一个主题画廊，社区还有一批以 npm 包形式发布的主题，从干净的编辑排版风格到深色的开发者大会风格都有。因为主题就是代码，所谓"自定义 Slidev 主题"意味着创建或 fork 一个主题包，配上你自己的版式、字体和配色——这比换一份样式表更费工夫，但可以完整地纳入版本控制。</p>
      <p><strong>Slidev 示例</strong>散落在文档和社区里，而不是集中在一个填空式的画廊。官网（sli.dev）本身就是用 Slidev 搭的，仓库里带有起步用的幻灯片，GitHub 上更满是真实的大会演讲，源码你都能读、能克隆。如果你在搜"slidev examples"，信噪比最高的就是官方演示和文档源码——它们几乎把每个功能都摆在了实处。对新手来说，坑和任何代码优先的工具一样：这里的"示例"是一个可克隆、可编辑的仓库，而不是一个你在界面里填空的模板。</p>
      <h2>把 Slidev 导出为 PowerPoint 与 PDF</h2>
      <p><strong>PDF 导出是一等公民、受官方支持，而且效果很好。</strong>Slidev 内置了 <code>slidev export</code> 命令（背后由 Playwright 驱动），能把每一页渲染成干净的、一页一张的 PDF——这是受支持的路径，导出的 PDF 和你的幻灯片长得一模一样。你也能用同样的方式导出成 PNG 图片。</p>
      <p><strong>而把 Slidev 导出为 PowerPoint，就是需要管理预期的地方了。</strong>Slidev 能生成 <code>.pptx</code>，但结果是<strong>有损的</strong>：每一页到了 PowerPoint 里都是一张扁平的图片，而不是可编辑的文本框、形状和版式。所以如果下游有人需要在 PowerPoint 里打开你的幻灯片并<em>编辑</em>它——重打一句标题、给图表换个样式、让某个要点重新排版——Slidev 导出的 <code>.pptx</code> 给不了他们这些。Slidev 生来就是要在浏览器里演示的，它可编辑的源是 Markdown 加 Vue，而不是 <code>.pptx</code>。在动手之前就明白这一点，能让你在同事哪天开口要"那份可编辑的 PowerPoint"时，免去一场难受的意外。</p>
      <h2>Slidev vs Marp vs reveal.js</h2>
      <p>Slidev 并不是把幻灯片写成代码的唯一途径。两个最接近的开源选择——也就是人们说"Slidev 替代品"时所指的——是 <strong>reveal.js</strong> 和 <strong>Marp</strong>。这里也该坦白：Open Design 处在一个完全不同的层次，而不是与它们正面竞争。</p>

      <p><a href="/blog/reveal-js/"><strong>reveal.js</strong></a> 是更老、更原始的那个同门：你用 HTML（或 Markdown）撰写，靠 data 属性接入过渡效果，拿 Slidev 的 Markdown 顺手换取更底层的 HTML 控制力。如果你想要最大的自由度、又不介意写标记语言，reveal.js 是自然之选。<a href="/blog/marp/"><strong>Marp</strong></a> 则走向另一个方向——<em>只</em>用 Markdown，刻意保持极简，没有组件——这让它在幻灯片以文字为主、你又完全不想操心框架时，成为最快的路径。Slidev 居于中间：以 Markdown 优先求速度，需要某一页多做点事时又能随手调用 Vue 组件。</p>
      <p>这三者都是代码优先的工具，骨架本质相同：你写幻灯片、在浏览器里演示，可编辑的 PowerPoint 输出本就不是重点。在它们之间做选择，主要是看你想要哪种撰写载体——Markdown 加 Vue、原始 HTML，还是纯 Markdown——而不是谁有、谁没有某项能力。</p>
      <h2>Open Design 的位置</h2>
      <p>这里是一条诚实的边界。Slidev 要求你去<strong>写</strong>这套幻灯片——每一页，用 Markdown 和 Vue。如果你想要手工撰写、纳入版本控制、带实时组件的幻灯片，这是优点，而且没有什么能替代它。</p>
      <p><a href="/">Open Design</a> 是另一个层次，而且它是我们做的产品，请带着这一点来读下面的内容。它是一个开源（Apache-2.0）、本地优先、自备 key 的 <strong>Agent-Native Design Workspace</strong>，位于你已经在用的编程 Agent——Claude Code、Codex 或 Cursor——<em>之外</em>。你不必手写 Markdown 和组件，而是描述你想要的幻灯片，Agent 就对着一套设计系统生成一份可编辑的，于是它一出来就符合品牌调性，并且保持可编辑，而不是冻结成一件渲染好的成品。Open Design 还提供了可供起步的第一方<a href="/plugins/templates/">幻灯片模板</a>。</p>
      <p>如果你要的就是把幻灯片当代码写、用 Vue 组件、掌握完整控制权，那它不是 Slidev 的替代品——那是 Slidev 的主场，它在那里表现出色。Open Design 适合的场景是：你宁愿给 Agent 下简报，也不想写标记语言；你希望输出在许多套幻灯片之间都匹配同一套设计系统；你希望最后拿到归你所有的可编辑文件。当这套幻灯片<em>就是</em>你正在写的 Markdown 时，去找 Slidev；当这套幻灯片只是 Agent 为你产出的众多符合品牌调性的输出之一时，去找一个工作空间。</p>
      <h2>常见问题</h2>
      <p><strong>Slidev 免费吗？</strong>免费。Slidev 在 MIT 许可证下开源——可免费商用，无需注册、没有水印、没有导出限制。你从自己的项目里自行运行它。</p>
      <p><strong>我该怎么使用 Slidev 主题？</strong>在幻灯片的 frontmatter 里设置主题（例如 <code>theme: seriph</code>），Slidev 就会安装并应用它。官方有一个画廊，npm 上还有社区主题。自定义主题意味着创建或 fork 一个主题包，配上你自己的版式、字体和配色——它是代码，不是你随手一换的样式表。</p>
      <p><strong>我能把 Slidev 导出成 PowerPoint 吗？</strong>Slidev 能导出 <code>.pptx</code>，但它是有损的——每一页都成了一张扁平图片，而不是可编辑的文本和形状。PDF 和 PNG 导出（通过 <code>slidev export</code>）才是干净的一等路径。如果你需要一份<em>可编辑的</em> PowerPoint，Slidev 不是对的源头。</p>
      <p><strong>Slidev、Marp 还是 reveal.js——我该用哪个？</strong>三者都是开源、代码优先、MIT 许可证。要 Markdown 加 Vue 组件（幻灯片需要交互时很棒）就选 Slidev，要原始 HTML 控制力就选 reveal.js，要最简单的纯 Markdown 幻灯片就选 Marp。它们都不以可编辑的 PowerPoint 输出为目标。</p>
      <p><strong>用 Slidev 需要会写代码吗？</strong>基本上，需要。Slidev 是一个开发者工具——你在文本编辑器和终端里，跟 Markdown、Vue 和一个开发服务器打交道，而不是一个拖拽式图形界面。如果你更愿意用大白话描述一套幻灯片，那正是像 Open Design 这样的 agent-native 工作空间要填补的空白。</p>
      <h2>结语</h2>
      <p>Slidev 配得上它的 47.6k 颗星：它是一种免费、MIT 许可证、开发者优先的方式，用 Markdown 做出好看的幻灯片，还带有货真价实的主题、热重载、实时 Vue 组件和干净的 PDF 导出。只是要心里有数——它是一个你在里面写东西的工具，而不是一个你点点点就能用的应用，"导出成可编辑的 PowerPoint"并不是它擅长的事。如果你想把幻灯片当代码来撰写，Slidev（或 <a href="/blog/reveal-js/">reveal.js</a> / <a href="/blog/marp/">Marp</a>）是个很好的归宿。如果你更愿意描述一套幻灯片、让你的 Agent 生成一份归你所有、可编辑、符合品牌调性的，那就该由一个 <a href="/">Agent-Native Design Workspace</a> 来接手了。</p>

      <h2>Slidev とは</h2>
      <p>Slidev はブラウザで動作し、デッキをドキュメントではなくプロジェクトとして扱います。単一の Markdown ファイルでスライドを書き、<code>---</code> で各ページを区切ります。そして開発者向けのツールこそが要点です——タイプしながらのホットリロード、行フォーカス付きの一流のコードハイライト、LaTeX や図表、ノート付きの発表者モード、そしてターミナルから起動する開発サーバー。素の Markdown が力尽きるところでは、<strong>Vue コンポーネント</strong>をスライドに直接埋め込めます——チャート、ライブカウンター、あるいは自作のインタラクティブなウィジェットを、箇条書きの隣にインラインで置けるのです。出力そのものが Web アプリ<em>である</em>ため、Web にできることは何でもスライド上で生きられます。</p>
      <p>トレードオフはその売り文句にそのまま書かれています。これは <strong>GUI ではなく開発者向けツール</strong>です。あなたのワークフローはドラッグ用のハンドルが並ぶキャンバスではなく、コードエディタと開発サーバーです——だからこそ、すでに Markdown と Web コンポーネントを好む開発者が手に取り、だからこそ非エンジニアははじき返されます。しかも本当に無料です。アカウント不要、ウォーターマークなし、エクスポート上限なし、席数課金なし。Markdown もその周りのプロジェクトも、すべてあなたのものです。</p>
      <h2>テーマと実例</h2>
      <p>Slidev をめぐる検索の多くは、実のところ、全スライドをゼロから設計せずに<em>見栄え</em>を良くしたい、というものです。そこで、各パーツがどう噛み合うのかを説明します。</p>
      <p><strong>Slidev のテーマ</strong>は組み込みの出発点であり、本当の強みです。frontmatter に 1 行書いてテーマを指定すると（たとえば <code>theme: seriph</code>）、Slidev がそれを読み込みます。公式のテーマギャラリーに加え、npm パッケージとして公開されたコミュニティ製のテーマも充実しており、クリーンな編集デザインからダークな開発者カンファレンス風まで揃っています。テーマはコードなので、「カスタム Slidev テーマ」とは、独自のレイアウト・フォント・カラーを備えたテーマパッケージを作るか fork することを意味します——スタイルシートを差し替えるより手はかかりますが、完全にバージョン管理できます。</p>
      <p><strong>Slidev の実例</strong>は、ひとつの穴埋め式ギャラリーではなく、ドキュメントとコミュニティの中にあります。公式サイト（sli.dev）自体が Slidev で作られており、リポジトリにはスターター用のデッキが含まれ、GitHub には本物のカンファレンス講演があふれていて、そのソースを読んでクローンできます。「slidev examples」を探しているなら、最も信頼できるのは公式デモとドキュメントのソースです——ほぼすべての機能が実際に使われた形で見られます。初心者にとっての落とし穴は、あらゆるコードファーストのツールと同じです。ここでの「実例」は、UI で穴を埋めるテンプレートではなく、クローンして編集するリポジトリなのです。</p>
      <h2>Slidev を PowerPoint と PDF にエクスポートする</h2>
      <p><strong>PDF エクスポートは一級の、公式サポートされた機能</strong>であり、しっかり動きます。Slidev には組み込みの <code>slidev export</code> コマンド（内部は Playwright）があり、各スライドをクリーンな 1 ページ 1 枚の PDF にレンダリングします——これがサポートされた経路で、PDF はデッキと寸分違わぬ見た目になります。同じ方法で PNG 画像へのエクスポートも可能です。</p>
      <p><strong>一方、Slidev を PowerPoint にエクスポートするとなると、期待値の調整が必要です。</strong>Slidev は <code>.pptx</code> を生成できますが、その結果は<strong>ロスあり</strong>です。各スライドは PowerPoint 上で、編集可能なテキストボックス・図形・レイアウトではなく、1 枚の平らな画像として届きます。ですから、後工程の誰かがあなたのデッキを PowerPoint で開いて<em>編集</em>する必要がある場合——見出しを打ち直す、チャートのスタイルを変える、箇条書きを流し直す——Slidev の <code>.pptx</code> ではそれができません。Slidev はブラウザから発表するために作られており、その編集可能なソースは <code>.pptx</code> ではなく Markdown と Vue です。始める前にこれを知っておけば、同僚が「編集できる PowerPoint がほしい」と言い出した日の、つらい不意打ちを避けられます。</p>
      <h2>Slidev vs Marp vs reveal.js</h2>
      <p>スライドをコードとして書く方法は Slidev だけではありません。最も近いオープンソースの選択肢——人々が「Slidev の代替」と言うときに指すもの——は <strong>reveal.js</strong> と <strong>Marp</strong> です。そして正直に言えば、Open Design は真っ向から競合するのではなく、まったく別のレイヤーに位置しています。</p>
      <table><thead><tr><th>ツール</th><th>執筆方法</th><th>編集可能な .pptx</th><th>テンプレート化</th><th>ライセンス</th></tr></thead><tbody><tr><td><strong>Slidev</strong></td><td>Markdown + Vue</td><td>不可（PPTX/PDF は画像）</td><td>Vue コンポーネント + テーマ</td><td>MIT</td></tr><tr><td><strong>reveal.js</strong></td><td>HTML または Markdown</td><td>不可（PDF→PPTX はロスあり画像）</td><td>テーマ = CSS/Sass；コミュニティテンプレート</td><td>MIT</td></tr><tr><td><strong>Marp</strong></td><td>Markdown のみ</td><td>不可（PDF/PPTX は画像）</td><td>Markdown テーマ（CSS）</td><td>MIT</td></tr><tr><td><strong>Open Design</strong></td><td>エージェントに説明する</td><td>可——デザインシステムに沿った編集可能な出力</td><td>ファーストパーティのデッキテンプレート</td><td>Apache-2.0</td></tr></tbody></table>
      <p><a href="/blog/reveal-js/"><strong>reveal.js</strong></a> は、より古く、より素朴な兄貴分です。HTML（または Markdown）で執筆し、data 属性でトランジションを組み込みます。Slidev の Markdown の書き心地を、より低レベルな HTML 制御と引き換えにするわけです。最大限の自由がほしく、マークアップを書くのを厭わないなら、reveal.js が自然な選択です。<a href="/blog/marp/"><strong>Marp</strong></a> は逆方向へ進みます——<em>Markdown のみ</em>、意図的にミニマルで、コンポーネントはなし——おかげで、スライドがほぼテキストで、フレームワークのことを一切考えたくないときには最速の道になります。Slidev はその中間に位置します。速さのために Markdown ファースト、そしてスライドがもう少し多くを必要とするときに Vue コンポーネントをすぐ呼び出せます。</p>
      <p>3 つとも、根本的な形は同じコードファーストのツールです。デッキを書き、ブラウザで発表し、編集可能な PowerPoint 出力は本来の狙いではありません。これらの間で選ぶのは、主にどの執筆面がほしいか——Markdown と Vue、素の HTML、あるいは素の Markdown——の問題であって、一方にあって他方にない機能の問題ではありません。</p>
      <h2>Open Design が位置づく場所</h2>
      <p>ここが正直な境界線です。Slidev はデッキを<strong>書く</strong>ことを求めます——すべてのスライドを、Markdown と Vue で。手で書き、バージョン管理され、ライブコンポーネントを備えたスライドがほしいなら、これは長所であり、それが目的なら何もこれに代わるものはありません。</p>
      <p><a href="/">Open Design</a> は別のレイヤーで、しかも私たちが作っている製品です。その点を踏まえて読んでください。これはオープンソース（Apache-2.0）、ローカルファースト、キー持ち込み方式の <strong>Agent-Native Design Workspace</strong> であり、あなたがすでに使っているコーディングエージェント——Claude Code、Codex、Cursor——の<em>外側</em>に位置します。Markdown とコンポーネントを手書きする代わりに、ほしいデッキを説明すると、エージェントがデザインシステムに沿って編集可能なものを生成します。だからブランドに沿った形で仕上がり、レンダリング済みの成果物として凍りつくのではなく、編集可能なまま保たれます。Open Design は、出発点として使えるファーストパーティの<a href="/plugins/templates/">デッキテンプレート</a>も提供しています。</p>
      <p>Vue コンポーネントと完全な制御でスライドをコードとして書きたいなら、これは Slidev の代わりにはなりません——それは Slidev の本拠地であり、そこでは実に見事です。Open Design が向くのは、マークアップを書くよりエージェントにブリーフを渡したい、多数のデッキにわたって出力をデザインシステムに合わせたい、そして最後に自分が所有する編集可能なファイルがほしい、という場合です。デッキが、あなたが書いている Markdown そのもの<em>である</em>ときは Slidev を。デッキが、エージェントがあなたのために生み出す数多くのブランド準拠の出力のひとつであるときは、ワークスペースを手に取ってください。</p>
      <h2>FAQ</h2>
      <p><strong>Slidev は無料ですか？</strong> はい。Slidev は MIT ライセンスのオープンソースです——商用利用も無料、アカウント不要、ウォーターマークなし、エクスポート上限なし。自分のプロジェクトから自分で実行します。</p>
      <p><strong>Slidev のテーマはどう使いますか？</strong> スライドの frontmatter でテーマを指定すると（たとえば <code>theme: seriph</code>）、Slidev がインストールして適用します。公式ギャラリーに加え、npm 上のコミュニティテーマがあります。カスタムテーマとは、独自のレイアウト・フォント・カラーを備えたテーマパッケージを作るか fork することを意味します——差し替えるスタイルシートではなく、コードです。</p>
      <p><strong>Slidev を PowerPoint にエクスポートできますか？</strong> Slidev は <code>.pptx</code> にエクスポートできますが、ロスありです——各スライドは編集可能なテキストや図形ではなく、平らな画像になります。PDF と PNG のエクスポート（<code>slidev export</code> 経由）が、クリーンで一級の経路です。<em>編集可能な</em> PowerPoint が必要なら、Slidev は適切なソースではありません。</p>
      <p><strong>Slidev、Marp、reveal.js——どれを使うべき？</strong> 3 つともオープンソースで、コードファーストで、MIT ライセンスです。Markdown と Vue コンポーネント（スライドにインタラクティブ性が必要なときに最適）なら Slidev、素の HTML 制御なら reveal.js、最もシンプルな Markdown のみのデッキなら Marp を選びましょう。どれも編集可能な PowerPoint 出力を狙っていません。</p>
      <p><strong>Slidev を使うにはコードを書ける必要がありますか？</strong> 事実上、必要です。Slidev は開発者向けツールです——ドラッグ＆ドロップの GUI ではなく、テキストエディタとターミナルで、Markdown、Vue、開発サーバーを扱います。デッキを平易な言葉で説明したいなら、それこそ Open Design のようなエージェントネイティブなワークスペースが埋める隙間です。</p>
      <h2>まとめ</h2>
      <p>Slidev は 47.6k のスターにふさわしいツールです。無料、MIT ライセンス、開発者ファーストで、Markdown から見栄えのするデッキを作れ、本物のテーマ、ホットリロード、ライブ Vue コンポーネント、そしてクリーンな PDF エクスポートを備えています。ただし覚悟のうえで飛び込みましょう——これはクリックして進むアプリではなく、その中で書くツールであり、「編集可能な PowerPoint へのエクスポート」は得意ではありません。デッキをコードとして執筆したいなら、Slidev（あるいは <a href="/blog/reveal-js/">reveal.js</a> / <a href="/blog/marp/">Marp</a>）は素晴らしい住処です。デッキを説明して、エージェントに、自分が所有する編集可能でブランド準拠のものを生成してほしいなら、そこからは <a href="/">Agent-Native Design Workspace</a> が引き継ぎます。</p>

      <h2>Slidev란</h2>
      <p>Slidev는 브라우저에서 실행되며, 덱을 문서가 아니라 프로젝트로 다룹니다. 단일 Markdown 파일에 슬라이드를 작성하고 <code>---</code>로 각 페이지를 구분하며, 개발자 도구가 바로 핵심입니다. 타이핑하는 대로 핫 리로드, 라인 포커스를 지원하는 일급 코드 하이라이트, LaTeX와 다이어그램, 노트가 있는 발표자 모드, 그리고 터미널에서 실행하는 개발 서버까지. 순수 Markdown이 한계에 부딪히는 지점에서는 <strong>Vue 컴포넌트</strong>를 슬라이드에 직접 임베드할 수 있습니다——차트, 실시간 카운터, 혹은 직접 만든 인터랙티브 위젯을 불릿 옆에 인라인으로 둘 수 있죠. 출력 자체가 웹 앱<em>이기</em> 때문에, 웹이 할 수 있는 것이라면 무엇이든 슬라이드 위에서 살아 있을 수 있습니다.</p>
      <p>절충점은 그 소개 문구에 그대로 적혀 있습니다. 이것은 <strong>GUI가 아니라 개발자 도구</strong>입니다. 당신의 워크플로는 드래그 핸들이 달린 캔버스가 아니라 코드 에디터와 개발 서버입니다——이미 Markdown과 웹 컴포넌트를 좋아하는 개발자가 이 도구를 집어 드는 이유가 바로 이것이고, 비개발자가 튕겨 나가는 이유도 바로 이것입니다. 게다가 진짜로 무료입니다. 계정도, 워터마크도, 내보내기 제한도, 좌석당 과금도 없습니다. Markdown과 그것을 둘러싼 프로젝트 전부가 당신의 것입니다.</p>
      <h2>테마와 예제</h2>
      <p>Slidev를 둘러싼 검색의 대부분은 사실 모든 슬라이드를 처음부터 디자인하지 않고도 <em>보기 좋게</em> 만들고 싶다는 것입니다. 그래서 각 조각이 어떻게 맞물리는지 짚어 보겠습니다.</p>
      <p><strong>Slidev 테마</strong>는 기본 제공되는 출발점이자 진짜 강점입니다. frontmatter에 한 줄로 테마를 지정하면(예: <code>theme: seriph</code>) Slidev가 그것을 가져옵니다. 공식 테마 갤러리에 더해, npm 패키지로 배포된 커뮤니티 테마도 충분히 많으며, 깔끔한 편집 디자인부터 어두운 개발자 콘퍼런스 스타일까지 다양합니다. 테마는 곧 코드이므로, "커스텀 Slidev 테마"란 자신만의 레이아웃, 폰트, 색상을 담은 테마 패키지를 만들거나 fork하는 것을 뜻합니다——스타일시트를 바꾸는 것보다는 손이 더 가지만, 완전히 버전 관리할 수 있습니다.</p>
      <p><strong>Slidev 예제</strong>는 하나의 빈칸 채우기 갤러리가 아니라 문서와 커뮤니티 속에 흩어져 있습니다. 공식 사이트(sli.dev) 자체가 Slidev로 만들어졌고, 저장소에는 시작용 덱이 들어 있으며, GitHub에는 소스를 읽고 클론할 수 있는 진짜 콘퍼런스 발표가 가득합니다. "slidev examples"를 찾고 있다면, 신호가 가장 강한 것은 공식 데모와 문서 소스입니다——거의 모든 기능이 실제로 쓰인 모습을 보여 주죠. 초보자에게 함정은 여느 코드 우선 도구와 같습니다. 여기서 "예제"란 UI로 빈칸을 채우는 템플릿이 아니라 클론해서 편집하는 저장소입니다.</p>
      <h2>Slidev를 PowerPoint와 PDF로 내보내기</h2>
      <p><strong>PDF 내보내기는 일급이자 공식 지원 기능</strong>이며 잘 작동합니다. Slidev에는 내장 <code>slidev export</code> 명령(내부적으로 Playwright 사용)이 있어 모든 슬라이드를 깔끔한 페이지당 한 장짜리 PDF로 렌더링합니다——지원되는 경로이며, PDF는 당신의 덱과 똑같아 보입니다. 같은 방식으로 PNG 이미지로도 내보낼 수 있습니다.</p>
      <p><strong>반면 Slidev를 PowerPoint로 내보내는 일은 기대치를 관리해야 하는 지점입니다.</strong> Slidev는 <code>.pptx</code>를 만들 수 있지만 그 결과는 <strong>손실이 있습니다</strong>. 각 슬라이드는 PowerPoint에서 편집 가능한 텍스트 상자, 도형, 레이아웃이 아니라 평평한 이미지 한 장으로 도착합니다. 그러니 후속 작업에서 누군가 당신의 덱을 PowerPoint로 열어 <em>편집</em>해야 한다면——제목을 다시 치고, 차트 스타일을 바꾸고, 불릿을 다시 흐르게 하고——Slidev의 <code>.pptx</code>는 그걸 해 주지 못합니다. Slidev는 브라우저에서 발표하도록 만들어졌고, 편집 가능한 소스는 <code>.pptx</code>가 아니라 Markdown과 Vue입니다. 시작하기 전에 이 점을 알아 두면, 동료가 "편집 가능한 PowerPoint 좀"이라고 요청하는 날의 괴로운 놀라움을 피할 수 있습니다.</p>
      <h2>Slidev vs Marp vs reveal.js</h2>
      <p>슬라이드를 코드로 쓰는 방법이 Slidev만 있는 것은 아닙니다. 가장 가까운 오픈소스 선택지——사람들이 "Slidev 대안"이라 할 때 가리키는 것——는 <strong>reveal.js</strong>와 <strong>Marp</strong>입니다. 그리고 솔직히 말하면, Open Design은 정면으로 경쟁하기보다 완전히 다른 층위에 자리합니다.</p>

Slidev는 그 중간에 자리합니다. 속도를 위해 Markdown 우선이면서, 어떤 슬라이드가 좀 더 해야 할 때는 Vue 컴포넌트를 바로 꺼내 쓸 수 있습니다.</p>
      <p>셋 다 근본 형태가 같은 코드 우선 도구입니다. 덱을 작성하고, 브라우저에서 발표하며, 편집 가능한 PowerPoint 출력은 애초에 목표가 아닙니다. 이들 사이에서 고르는 일은 주로 어떤 작성 표면을 원하느냐——Markdown과 Vue, 순수 HTML, 아니면 순수 Markdown——의 문제이지, 한쪽에는 있고 다른 쪽에는 없는 기능의 문제가 아닙니다.</p>

If you write code and you'd rather not open PowerPoint, you've probably run into **Slidev**. It bills itself as "Presentation Slides for Developers," and that's exactly what it is: you write each slide in Markdown, drop in Vue components and code blocks where you need them, and present the whole thing as a web page — with hot reload, syntax highlighting, presenter notes, and interactive embeds built in. With 47.6k stars on GitHub and an MIT license, it's one of the go-to answers to "how do I make a good-looking deck from Markdown."

This guide is an honest map of it. What Slidev actually is, how themes and examples fit together, how exporting to PDF and PowerPoint really behaves (there's a catch), how it stacks up against the other open-source contenders, and — the part most write-ups skip — where a tool like this stops being the right one and something else takes over. If you're weighing your options for an [AI-and-agent-assisted deck workflow](/blog/claude-ppt-skills/), it helps to know precisely what Slidev is and isn't.

## What Slidev is

Slidev runs in the browser and treats your deck as a project, not a document. You author slides in a single Markdown file, separating them with `---`, and the developer tooling is the point: hot reload as you type, first-class code highlighting with line focus, LaTeX and diagrams, presenter mode with notes, and a dev server you run from your terminal. Where plain Markdown runs out of road, you embed **Vue components** directly in a slide — so a chart, a live counter, or your own interactive widget can sit inline next to your bullet points. Because the output *is* a web app, anything the web can do can live on a slide.

The trade-off is right there in the pitch: this is a **developer tool, not a GUI**. Your workflow is a code editor and a dev server, not a canvas with drag handles — which is exactly why developers who already like Markdown and web components reach for it, and exactly why a non-coder will bounce off it. It's also genuinely free: no account, no watermark, no export cap, no per-seat pricing. You own the Markdown and the project around it.

## Themes & examples

Most searches around Slidev are really about making it *look* good without designing every slide from scratch, so here's how the pieces fit.

**Slidev themes** are the built-in starting point, and they're a real strength. You set a theme with a single line in your frontmatter (`theme: seriph`, for example), and Slidev pulls it in. There's an official gallery of themes plus a healthy set of community ones published as npm packages, ranging from clean editorial looks to dark developer-conference styles. Because a theme is code, a "custom Slidev theme" means creating or forking a theme package with your own layouts, fonts, and colors — more involved than swapping a stylesheet, but fully version-controllable.

**Slidev examples** live in the docs and the community rather than in one fill-in gallery. The official site (sli.dev) is itself built with Slidev, the repository ships starter decks, and GitHub is full of real conference talks whose source you can read and clone. If you're searching for "slidev examples," the highest-signal ones are the official demos and the docs source — they show nearly every feature in place. The catch for beginners is the same as with any code-first tool: an "example" here is a repository to clone and edit, not a template you fill in through a UI.

## Exporting Slidev to PowerPoint & PDF

**PDF export is a first-class, supported feature** and it works well. Slidev has a built-in `slidev export` command (backed by Playwright) that renders every slide to a clean, page-per-slide PDF — the supported path, and the PDF looks exactly like your deck. You can also export to PNG images the same way.

**Exporting Slidev to PowerPoint is where expectations need managing.** Slidev can produce a `.pptx`, but the result is **lossy**: each slide arrives in PowerPoint as a flat image, not as editable text boxes, shapes, and layouts. So if someone downstream needs to open your deck in PowerPoint and *edit* it — retype a headline, restyle a chart, reflow a bullet — the Slidev `.pptx` won't give them that. Slidev is built to be presented from the browser, and its editable source is Markdown-plus-Vue, not `.pptx`. Knowing this before you start saves a painful surprise the day a colleague asks for "the editable PowerPoint."

## Slidev vs Marp vs reveal.js

Slidev isn't the only way to write slides as code. The two closest open-source options — the ones people mean by "Slidev alternative" — are **reveal.js** and **Marp**, and it's worth being honest that Open Design sits in a different layer entirely rather than competing head-on.

| Tool | Authoring | Editable .pptx | Templating | License |
| --- | --- | --- | --- | --- |
| **Slidev** | Markdown + Vue | No (PPTX/PDF as images) | Vue components + themes | MIT |
| **reveal.js** | HTML or Markdown | No (PDF→PPTX is lossy images) | Themes = CSS/Sass; community templates | MIT |
| **Marp** | Markdown only | No (PDF/PPTX as images) | Markdown themes (CSS) | MIT |
| **Open Design** | Describe it to your agent | Yes — editable output against a design system | First-party deck templates | Apache-2.0 |

[**reveal.js**](/blog/reveal-js/) is the older, more raw sibling: you author in HTML (or Markdown) and wire in transitions with data-attributes, trading Slidev's Markdown ergonomics for lower-level HTML control. If you want maximum freedom and don't mind writing markup, reveal.js is the natural pick. [**Marp**](/blog/marp/) goes the other direction — Markdown *only*, deliberately minimal, no components — which makes it the fastest path when your slides are mostly text and you never want to think about a framework. Slidev sits in the middle: Markdown-first for speed, with Vue components on tap when a slide needs to do more.

All three are code-first tools with the same fundamental shape: you write the deck, you present in a browser, and editable-PowerPoint output isn't really the point. Choosing between them is mostly about which authoring surface you want — Markdown-plus-Vue, raw HTML, or plain Markdown — not about capabilities one has and the others lack.

## Where Open Design fits

Here's the honest boundary. Slidev asks you to **write** the deck — every slide, in Markdown and Vue. That's a feature if you want hand-authored, version-controlled slides with live components, and nothing here replaces it if that's your goal.

[Open Design](/) is a different layer, and it's a product we make, so read this with that in mind. It's an open-source (Apache-2.0), local-first, bring-your-own-key **Agent-Native Design Workspace** that sits *outside* the coding agent you already use — Claude Code, Codex, or Cursor. Reach for Slidev when the deck *is* the Markdown you're writing; reach for a workspace when the deck is one on-brand output among many that an agent produces for you.

## FAQ

**Is Slidev free?** Yes. Slidev is open source under the MIT license — free to use commercially, with no account, no watermark, and no export limits. You run it yourself from your own project.

**How do I use Slidev themes?** Set a theme in your slide frontmatter (for example `theme: seriph`) and Slidev installs and applies it. There's an official gallery plus community themes on npm. A custom theme means creating or forking a theme package with your own layouts, fonts, and colors — it's code, not a stylesheet you swap.

**Can I export Slidev to PowerPoint?** Slidev can export to `.pptx`, but it's lossy — each slide lands as a flat image, not editable text and shapes. PDF and PNG export (via `slidev export`) is the clean, first-class path. If you need an *editable* PowerPoint, Slidev isn't the right source.

**Slidev vs Marp vs reveal.js — which should I use?** All three are open-source, code-first, MIT-licensed. Pick Slidev for Markdown-plus-Vue components (great when slides need to be interactive), reveal.js for raw HTML control, and Marp for the simplest Markdown-only decks. None targets editable-PowerPoint output.

**Do I need to know how to code to use Slidev?** Effectively, yes. Slidev is a developer tool — you work in a text editor and a terminal with Markdown, Vue, and a dev server, not a drag-and-drop GUI. If you'd rather describe a deck in plain language, that's the gap an agent-native workspace like Open Design fills.
